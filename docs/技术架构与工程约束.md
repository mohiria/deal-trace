## 2. 架构目标

本系统采用前后端分离架构，面向中小型 B2B 销售团队的内部 CRM 使用场景。

技术架构目标：

1. 保持 MVP 架构简单、稳定、可维护。
2. 将核心业务规则放在后端强校验，前端仅做交互和体验层校验。
3. 使用数据库约束和后端事务兜底关键一致性。
4. 为 OpenSpec 变更包拆分数据模型、API、页面和实现任务提供明确边界。
5. 避免在架构文档中过早展开完整数据表、接口字段和页面细节。

---

## 3. 总体架构

系统采用单仓库 Monorepo 管理前端、后端、文档和 OpenSpec 规格产物。

推荐目录结构：

```text
deal-trace/
  frontend/
  backend/
  docs/
  openspec/
    specs/
    changes/
    explorations/
```

职责划分：

| 模块 | 职责 |
| :--- | :--- |
| `frontend/` | Vue 前端应用，负责页面、交互、路由、前端状态和 API 调用。 |
| `backend/` | Spring Boot 后端应用，负责认证、权限、业务规则、事务、数据访问和 API。 |
| `docs/` | PRD、技术架构、设计说明等人工维护文档。 |
| `openspec/specs/` | OpenSpec 归档后的稳定系统行为规格，描述当前系统如何工作。 |
| `openspec/changes/` | OpenSpec 变更包目录，每个变更包含 proposal、delta specs、design 和 tasks。 |
| `openspec/explorations/` | OpenSpec 探索记录，用于不确定问题的前置分析。 |

说明：

1. 根目录的 `OpenSpec/` 下载源码目录不是本项目业务规格目录，可以删除。
2. 根目录的 `vibe-coding-qa/` 下载资料目录不是本项目测试产物目录，后续通过正式安装的 skill/reference 使用。
3. 本项目业务规格使用根目录小写 `openspec/`。

## 4. 技术选型

### 4.1 前端技术栈

| 类型 | 选型 |
| :--- | :--- |
| 框架 | Vue 3 |
| 语言 | TypeScript |
| 构建工具 | Vite |
| UI 组件库 | Arco Design Vue |
| 路由 | Vue Router |
| 状态管理 | Pinia |
| HTTP 客户端 | Axios |

前端定位：

1. 负责表单校验、页面状态、权限入口显隐和用户交互。
2. 不作为权限和业务规则的最终可信边界。
3. 所有关键权限、状态流转、查重和并发控制必须由后端再次校验。

### 4.2 后端技术栈

| 类型 | 选型 |
| :--- | :--- |
| 语言 | Java 24.0.1 |
| 构建工具 | Maven 3.9.16 |
| 框架 | Spring Boot 4.0.6 |
| 安全框架 | Spring Security |
| ORM / 数据访问 | MyBatis Plus 3.5.16（`mybatis-plus-spring-boot4-starter`） |
| 密码存储 | BCrypt |
| API 风格 | RESTful JSON |

后端定位：

1. 负责认证、授权、业务规则、事务和数据一致性。
2. 所有写操作必须在后端校验当前用户权限。
3. 公海认领、归属变更、赢单、流失、重复线索创建等关键操作必须由后端事务兜底。

### 4.3 数据库与迁移

| 类型 | 选型 |
| :--- | :--- |
| 数据库 | MySQL 8.4 LTS |
| 存储引擎 | InnoDB |
| 数据库迁移 | Flyway |

数据库使用原则：

1. 关键唯一性使用数据库唯一索引兜底。
2. 金额字段使用精确数值类型，不使用浮点类型。
3. 时间字段统一由后端生成和写入。
4. 数据库结构由后续 OpenSpec 变更包中的 design 和相关规格详细定义。

## 5. 认证与权限

### 5.1 登录认证

MVP 采用邮箱和密码登录。

认证方案：

1. 登录成功后由后端签发 JWT。
2. 前端请求 API 时通过 `Authorization: Bearer <token>` 携带访问令牌。
3. 后端通过 Spring Security 校验 JWT。
4. 密码使用 BCrypt 哈希存储。
5. MVP 不包含 Sales 自助修改密码、找回密码和邀请邮件激活能力。

### 5.2 初始 Admin

系统预置一个初始 Admin 账号，用于首次进入系统。

约束：

1. 初始 Admin 的邮箱和初始密码由部署配置提供。
2. 初始 Admin 可创建 Sales 账号。
3. MVP 阶段 Admin 仅创建 Sales 账号，不支持创建多个 Admin。

### 5.3 权限模型

系统角色：

1. Admin。
2. Sales。

权限原则：

1. Admin 可查看全部客户、业务线索、进度跟踪、系统日志、合同记录和全局指标。
2. Sales 可搜索全部客户主体。
3. Sales 只能查看公海业务线索的脱敏信息和自己名下业务线索的完整信息。
4. Sales 不可查看其他 Sales 名下业务线索详情。
5. 停用 Sales 不可登录、不可认领公海线索、不可被分配或转移新线索。
6. 权限最终由后端强校验，前端权限控制仅用于减少无效入口和提升体验。

## 6. API 设计约束

### 6.1 API 风格

后端提供 RESTful JSON API。

基础约束：

1. API 路径使用资源化命名。
2. 请求和响应使用 JSON。
3. 前后端通过 OpenAPI 3 描述接口契约。
4. API 详细路径、请求字段、响应字段由后续 OpenSpec 变更包中的 design 和相关规格定义。

### 6.2 统一响应结构

普通响应建议结构：

```json
{
  "code": "SUCCESS",
  "message": "OK",
  "data": {}
}
```

分页响应建议结构：

```json
{
  "code": "SUCCESS",
  "message": "OK",
  "data": {
    "items": [],
    "total": 100,
    "page": 1,
    "pageSize": 20
  }
}
```

### 6.3 错误码类别

错误码由后续 API Spec 细化，但至少需要覆盖以下类别：

1. `VALIDATION_ERROR`：字段校验失败。
2. `UNAUTHORIZED`：未登录或 token 无效。
3. `FORBIDDEN`：无权限访问或操作。
4. `NOT_FOUND`：资源不存在。
5. `DUPLICATE_CUSTOMER`：客户重复。
6. `DUPLICATE_ACTIVE_LEAD`：同类进行中业务线索重复。
7. `DUPLICATE_WON_LEAD`：同类已赢单业务线索重复。
8. `LEAD_ALREADY_CLAIMED`：公海线索已被认领。
9. `LEAD_ENDED_READONLY`：已结束线索只读。
10. `ACCOUNT_DISABLED`：账号已停用。

## 7. 数据与业务一致性约束

### 7.1 数据模型边界

PRD v3.3 已确认核心业务对象：

1. 用户。
2. 客户。
3. 业务线索。
4. 进度跟踪。
5. 系统日志。
6. 合同记录。

完整数据表、字段类型、索引、外键、枚举存储方式由 OpenSpec 变更包中的 design 设计，并由相关行为规格约束。

### 7.2 必须数据库兜底的规则

后续数据模型至少需要考虑以下数据库约束：

1. 登录邮箱全局唯一。
2. 客户统一社会信用代码全局唯一。
3. 客户名称按规范化结果不可完全重复。
4. 每条业务线索最多一条合同记录。

同一自然年度、同一客户、同一业务类型的线索重复规则涉及状态判断，主要由后端事务校验实现；如数据库索引参与约束，需要由数据模型 Spec 明确实现策略。

### 7.3 必须后端事务兜底的操作

以下操作必须由后端事务保证一致性：

1. 创建业务线索时的同类线索查重。
2. 公海认领。
3. Admin 分配、回收、转移业务线索。
4. Sales 退回公海。
5. 阶段变更。
6. 标记赢单并生成合同记录。
7. 标记流失。
8. 停用账号。

并发场景中，前端按钮禁用不能作为一致性保障。

## 8. 关键业务规则实现策略

### 8.1 客户唯一性

客户唯一性以统一社会信用代码为主，同时客户名称也不可完全重复。

实现原则：

1. 统一社会信用代码保存和查重前去除首尾空格，并统一转为大写。
2. 统一社会信用代码需符合 18 位统一社会信用代码标准格式，并通过校验位校验。
3. 客户名称保存和查重前去除首尾空格。
4. 客户名称去除首尾空格后完全一致时，不允许创建。

### 8.2 业务线索查重

同一自然年度、同一客户、同一业务类型，视为同类业务线索。

实现原则：

1. 存在进行中同类线索时，不允许新建。
2. 存在已赢单同类线索时，不允许新建。
3. 仅存在已流失同类线索时，允许新建，并提示历史流失线索。
4. 业务线索自然年度由创建时间生成，不允许人工修改。

### 8.3 公海认领并发

公海认领必须后端兜底。

认领成功条件：

1. 业务线索当前归属为空。
2. 业务线索未结束。
3. 当前 Sales 为启用状态。

认领失败时返回明确错误码，例如 `LEAD_ALREADY_CLAIMED` 或 `FORBIDDEN`。

### 8.4 结束状态只读

已赢单和已流失为结束状态。

结束状态业务线索：

1. 不可新增进度跟踪。
2. 不可修改阶段。
3. 不可退回公海。
4. 不可分配、回收或转移。
5. 不可重复标记赢单或流失。

### 8.5 系统日志与进度跟踪

进度跟踪和系统日志是两个独立对象。

进度跟踪：

1. 由有权限用户手动新增。
2. 记录销售推进内容。
3. 新增后不可编辑、不可删除。

系统日志：

1. 由系统自动生成。
2. 记录操作审计。
3. 仅可查看，不可编辑、不可删除。

## 9. 通用字段与格式约束

### 9.1 时间

时间字段原则：

1. 创建时间、更新时间、跟踪时间、操作时间由后端生成。
2. Dashboard 中“今日”“本月”按照系统服务端时间计算。
3. 具体时区策略由后续技术实现 Spec 明确，前后端需要保持一致。

### 9.2 金额

金额字段原则：

1. 合同金额单位为元。
2. 合同金额必须大于 0。
3. 合同金额最多两位小数。
4. 合同金额展示时使用千分位格式。
5. 存储使用精确数值类型。

### 9.3 枚举

业务类型固定枚举：

1. BIM咨询。
2. BIM培训。
3. 定制开发。

线索阶段固定枚举：

1. 未触达。
2. 初步沟通。
3. 方案报价。
4. 商务谈判。
5. 已赢单。
6. 已流失。

流失原因固定枚举：

1. 价格过高。
2. 选择竞品。
3. 无明确需求。
4. 联系不上。
5. 其他。

枚举的数据库存储方式由数据模型 Spec 明确。

### 9.4 联系电话

联系电话录入校验：

1. 手机号支持中国大陆 11 位手机号。
2. 座机支持常见区号格式，例如 `010-12345678`、`0571-12345678`、`0571-12345678-123`。
3. MVP 不支持海外号码。

联系电话脱敏：

1. 11 位手机号展示前三后四，例如 `138****5678`。
2. 非 11 位电话保留前 3 位和后 4 位，中间用 `****`。
3. 长度不足 8 位的电话，仅展示后 2 位，其余用 `*`。

## 10. 前端工程约束

前端实现原则：

1. 使用 Vue Router 管理页面路由。
2. 使用 Pinia 管理登录用户、权限、基础字典和必要页面状态。
3. 使用 Axios 封装 API 请求，统一处理 token、错误响应和登录失效。
4. 表单层做即时校验，但不替代后端校验。
5. 角色权限影响页面入口、按钮显隐和操作可用状态。
6. 所有越权操作必须以后端返回结果为准。
7. 不引入 Tailwind CSS 作为主样式体系，避免与 Arco Design Vue 主题体系冲突。

建议页面模块：

1. 登录页。
2. Dashboard。
3. 客户管理。
4. 公海线索。
5. 我的线索。
6. 线索详情。
7. 用户管理。
8. 合同记录。

页面细节、路由路径、表格字段、表单字段由 OpenSpec 变更包中的 design 和前端行为规格定义。

### 10.1 前端设计规范

前端设计风格参考 monday.com 的工作台式信息组织和轻量交互体验，但不复刻 monday.com 品牌、Logo、商标、专有图形或完全一致的视觉资产。

DealTrace 应形成自身的 CRM 工作台风格：

1. 整体视觉明亮、轻量、现代。
2. 信息密度适中，优先服务销售日常高频操作。
3. 列表、筛选、状态标签和快捷操作应清晰可扫读。
4. 状态使用彩色标签区分，但避免过度装饰。
5. 页面结构以工作台为主，不使用营销站式大面积视觉布局。
6. 交互反馈保持轻量，避免复杂动画影响效率。

Arco Design Vue 组件使用建议：

1. 列表页优先使用 Table。
2. 状态展示优先使用 Tag。
3. 操作区优先使用 Button、Dropdown、Modal。
4. 筛选区优先使用 Form、Input、Select、DatePicker。
5. 详情页优先使用 Tabs、Descriptions、Timeline。
6. Dashboard 优先使用 Card、Statistic。
7. 线索阶段可使用 Tag 或 Steps 表达。

后续 OpenSpec 生成前端相关规格和设计时，必须遵循 Arco Design Vue 组件体系和上述设计规范。

## 11. 后端工程约束

后端建议分层：

1. Controller：处理 HTTP 请求和响应。
2. Application / Service：承载业务流程、权限校验和事务。
3. Domain / Model：承载业务对象和枚举。
4. Mapper：负责数据库访问。
5. Security：负责认证、JWT 和权限上下文。
6. Common：统一响应、错误码、异常处理、工具类。

后端实现原则：

1. 写操作统一在 Service 层做权限和业务规则校验。
2. 关键写操作使用事务。
3. 系统日志由后端在业务操作成功后写入。
4. 业务错误使用统一错误码返回。
5. 不允许仅依赖前端控制实现权限或状态限制。

## 12. 测试与质量约束

后续测试设计应覆盖：

1. 权限隔离。
2. 客户唯一性。
3. 业务线索查重。
4. 公海认领并发。
5. 分配、回收、转移。
6. 阶段变更。
7. 赢单、流失。
8. 进度跟踪不可编辑删除。
9. 系统日志生成。
10. Dashboard 统计口径。

建议测试层级：

1. 后端单元测试：校验工具、业务规则、枚举转换。
2. 后端集成测试：API、事务、数据库约束、权限。
3. 前端组件/页面测试：关键表单、权限入口、列表操作。
4. E2E 测试：登录、客户创建、线索创建、公海认领、赢单/流失主链路。

后端集成测试建议使用 Testcontainers 运行 MySQL 8.4 LTS。

## 13. OpenSpec 与 vibe-coding-qa 工作流边界

### 13.1 OpenSpec 职责

OpenSpec 用于管理系统行为规格和变更包。

职责：

1. 将 PRD v3.3 和本技术架构文档转换为可实现、可验证的行为规格。
2. 使用 `openspec/changes/` 管理每个变更的 proposal、delta specs、design 和 tasks。
3. 使用 `openspec/specs/` 存放归档后的稳定系统行为规格。
4. 在归档变更时，将 delta specs 同步到稳定 specs。

OpenSpec 的 `spec.md` 应写行为契约：

1. Requirement。
2. Scenario。
3. Given / When / Then。

`spec.md` 不写完整数据库表结构、代码类名、测试文件路径或详细实现步骤。

### 13.2 初始 MVP 变更包

初始 MVP 作为第一个 OpenSpec change 管理。

推荐变更包：

```text
openspec/changes/bootstrap-dealtrace-mvp/
  proposal.md
  design.md
  tasks.md
  specs/
    auth-account/spec.md
    customer/spec.md
    lead/spec.md
    permission/spec.md
    progress-log/spec.md
    contract/spec.md
    dashboard/spec.md
    frontend-workbench/spec.md
  qa/
    lightweight-test-design.md
    regression-impact-analysis.md
    qa-test-report.md
```

行为规格按业务域拆分：

| 规格域 | 内容 |
| :--- | :--- |
| `auth-account` | 登录、初始 Admin、账号启用停用、密码和角色。 |
| `customer` | 客户主体、统一社会信用代码、客户搜索和唯一性。 |
| `lead` | 业务线索创建、查重、公海、认领、阶段、结束状态。 |
| `permission` | Admin/Sales 权限隔离、客户主体搜索、私海访问限制。 |
| `progress-log` | 进度跟踪和系统日志行为。 |
| `contract` | 赢单、合同记录、金额和签订日期规则。 |
| `dashboard` | 指标口径、Admin/Sales 统计范围。 |
| `frontend-workbench` | 工作台式前端体验、主要页面、Arco 组件约束。 |

### 13.3 vibe-coding-qa 职责

vibe-coding-qa 用于 TDD/QA 流程，不用于存放稳定业务规格。

职责：

1. 在每个 OpenSpec task 进入实现前，生成轻量测试设计。
2. 将测试点追溯到 PRD、OpenSpec spec、design、API contract 或数据模型。
3. 为适合 TDD 的单元/API/集成行为建立 Red-Green-Refactor 证据。
4. 为关键用户路径设计 E2E 场景。
5. 记录测试执行、覆盖证据、回归范围、阻塞项和剩余风险。

QA 产物放在对应 OpenSpec change 的 `qa/` 目录下：

```text
openspec/changes/<change-name>/qa/
  lightweight-test-design.md
  regression-impact-analysis.md
  qa-test-report.md
```

`qa/` 目录只放测试设计和 QA 报告，不放自动化测试脚本。

### 13.4 测试代码位置

自动化测试脚本放在真实工程测试目录：

```text
backend/src/test/...
frontend/src/**/*.test.ts
frontend/tests/e2e/...
```

测试设计和测试代码的关系：

1. `qa/lightweight-test-design.md` 记录测试点、测试层级和覆盖目标。
2. 自动化测试文件实现这些测试点。
3. `qa/qa-test-report.md` 记录 Red/Green、执行命令、覆盖证据和剩余风险。

### 13.5 本技术架构文档不负责的内容

本技术架构文档不负责详细定义以下内容：

1. 完整数据库表结构。
2. 完整 API 请求和响应字段。
3. 前端页面字段、布局和交互细节。
4. 详细测试用例。
5. 具体实现代码结构。

这些内容交由 OpenSpec change 的 `design.md`、`tasks.md`、delta specs，以及后续真实代码和测试实现承接。

### 13.6 权威来源和冲突处理

OpenSpec 拆分时必须以以下文档作为上游依据：

1. `产品需求文档 (PRD)：商迹 (DealTrace).md`
2. `docs/技术架构与工程约束.md`

如 OpenSpec 输出与 PRD 或本技术架构文档冲突，以 PRD 和本技术架构文档为准，并进入需求冲突确认流程。

如 vibe-coding-qa 测试设计与 PRD、OpenSpec spec、API contract、数据模型或既有测试发生冲突，必须使用 Requirement Conflict Gate，不得擅自修改期望行为或削弱测试断言。
