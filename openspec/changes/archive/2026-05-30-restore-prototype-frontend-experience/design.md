## Context

`prototype/dealtrace-workbench.html` 定义了比当前 Vue 实现更完整的 CRM 工作台体验。当前前端已经具备业务基础：登录后的外壳、Dashboard 指标、线索 store、线索详情面板、客户创建、线索查重、账号管理和按角色显示入口。主要缺口在视觉一致性、抽屉生命周期和真实交互状态的呈现。

既有约束：

- 保持当前后端 API、数据库 schema、Pinia store、路由结构和权限模型不变。
- 使用 Vue + Arco Design Vue + 既有 `--dt-*` token；不引入 Tailwind 或新的设计系统。
- 中文文案保持 UTF-8，并通过浏览器渲染结果验证可读性。
- `prototype/dealtrace-workbench.html` 作为视觉和交互参考，不作为可直接复制到生产页的静态展示区。
- 抽屉内不实现后端系统日志读取，因为当前没有对应读接口。

## Goals / Non-Goals

**Goals:**

- 让受保护工作区外壳、销售工作台、线索抽屉、客户管理页和用户管理页形成同一套原型风格体验。
- 保证工作台抽屉默认隐藏，用户选择线索后打开，支持显式关闭，关闭后清空选中行。
- 尽量复用当前 store 和 `LeadDetailPanel` 的业务逻辑，避免重新实现线索行为。
- 将原型中的关键交互示例转成真实页面状态和表单/弹窗反馈。
- 用聚焦测试覆盖 tab、筛选、抽屉打开/关闭、关键状态渲染和中文可读性风险。

**Non-Goals:**

- 不修改后端接口、schema 或 API client 契约。
- 不新增系统日志抽屉读取能力。
- 不实现导出、高级表格分页或服务端排序。
- 不替换 Arco Design Vue。
- 不在生产页面加入静态 `state-lab` 展示区。

## Decisions

### D1：把原型作为参考层，而不是整页搬运

当前应用有真实路由、登录态、角色权限和接口数据。原型是单 HTML 组合页，把外壳、Dashboard、抽屉和状态示例放在同一视口。实现时提取布局、间距、颜色、信息密度和交互意图，但保留当前 Vue 的路由和组件边界。

备选方案是把原型结构直接塞进 `DashboardView.vue`。这会重复外壳职责，绕开现有路由，并把真实交互示例变成静态标本，因此不采用。

### D2：在 `AppShell.vue` 还原外壳视觉

外壳应承载原型中的侧边栏风格：品牌标识、导航分组、当前项状态和提醒卡片。导航仍使用 `visibleSections(auth.role)` 和当前 `RouterLink`，保证角色入口不被视觉改动影响。

备选方案是只改工作台内部。这样首屏会同时出现旧外壳和新工作台风格，不利于整体体验一致性。

### D3：工作台数据源不变，三类 tab 做前端视图投影

工作台继续通过 `fetchDashboard()` 加载四项指标，通过现有 leads store 加载名下/全部线索和公海线索。三个 tab 的语义：

- `我的线索`：Sales 使用 `myLeads`；Admin 可继续沿用当前“全部线索”的解释。
- `公海线索`：使用 `pool`。
- `本月结束`：从当前已加载的名下/全部线索中筛选当前自然月结束的赢单或流失线索。

这样可以还原原型 tab 结构，同时避免新增后端接口。

### D4：抽屉生命周期由工作台控制

`DashboardView.vue` 持有 `activeLeadId`。`activeLeadId` 为 `null` 时抽屉不存在；用户点击线索后设置 id 并打开抽屉；用户关闭抽屉时重新置为 `null`。`LeadDetailPanel` 可以承载抽屉内容和关闭入口的展示上下文，但表格选中态应由工作台统一控制。

备选方案是一直保留右侧 sticky 面板。用户已经明确指出抽屉默认不应显示，因此不采用。

### D5：`LeadDetailPanel` 按原型调整表现，但继续共享业务逻辑

详情面板保留现有写入权限、后端兜底和错误处理逻辑，同时把表现调整为原型风格：抽屉标题、阶段轨道、字段网格、操作区、进度时间线和结束线索只读态。独立线索详情页仍可复用同一组件，页面包装可以不同，但业务逻辑不分叉。

系统日志展示暂不实现，除非后续新增后端读取能力。

### D6：把状态示例映射到真实客户/用户/线索流程

原型中的 `state-lab` 应指导真实页面状态：

- 客户新建中的重复客户和 USCI 校验反馈。
- 线索新建中的查重警告或阻断反馈。
- 结束线索在详情/抽屉中的只读表现。
- 赢单/流失弹窗的结构和校验反馈。
- Admin/Sales 操作差异在真实按钮和表格中体现。

备选方案是在 Dashboard 下方渲染静态 `state-lab`。这会污染生产 UI，也不能改善真实流程，因此不采用。

### D7：视觉变更需要浏览器可见验证

这次变更以视觉和交互还原为主，除单元/组件测试外，实施时应尽量启动前端并用非无头浏览器检查工作台、抽屉、客户/用户页面和中文渲染。终端里的中文乱码可能只是 PowerShell 编码显示问题，不能单独作为源码损坏依据。

## Risks / Trade-offs

- **原型字段与当前数据结构不一致** -> 仅展示已有字段或可靠派生值；没有后端字段时不伪造“预计金额”等业务数据。
- **共享 `LeadDetailPanel` 改动影响独立详情页** -> 保留详情行为测试，避免业务逻辑分叉。
- **终端输出中文乱码导致误判** -> 同时检查文件编码和浏览器渲染结果。
- **大范围样式改动影响无关页面** -> 样式限定在外壳、工作台、客户页、用户页和共享 token，不做无边界全局覆盖。
- **静态原型示例被误当成真实功能** -> 只在已有业务逻辑支撑的真实流程里实现对应状态。
