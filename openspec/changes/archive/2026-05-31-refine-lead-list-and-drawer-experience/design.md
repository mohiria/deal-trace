## Context

当前前端已完成原型风格的基础还原，但新增线索入口仍分散在页面跳转和客户页内部弹窗中；我的线索、公海线索、客户管理、用户管理列表体验不一致，且缺少统一搜索和分页。销售工作台抽屉的阶段轨道展示了未来阶段，关闭按钮与顶部内容距离较近；工作区右上角还展示角色，指标卡右上角 spark 图形视觉负担较重。

本轮只调整前端体验。后端接口、权限裁决、数据库结构、线索状态机和 Docker 部署均保持不变。

## Goals / Non-Goals

**Goals:**
- 统一新增线索入口，使工作台、今日提醒、我的线索、公海线索、客户管理复用同一套新建线索表单、查重和提交逻辑。
- 统一真实列表页搜索和标准分页，保持工作台风格的紧凑表格体验。
- 优化工作台抽屉阶段展示、关闭按钮位置、桌面宽度和小屏布局。
- 优化顶部人员信息和四个指标卡表现。

**Non-Goals:**
- 不新增后端分页、后端线索搜索或用户搜索接口。
- 不改变客户搜索接口、线索创建接口、查重接口、认领接口和账号管理接口。
- 不改变角色权限、字段可见性、线索状态迁移和 closed lead 只读规则。
- 不引入 Tailwind 或新的 UI 依赖。

## Decisions

1. **新建线索入口复用组件化实现**
   - 提取 `CreateLeadModal`，承载现有客户选择、业务类型、查重、历史流失提示、归属选项、字段校验和 `createLead` 提交流程。
   - 工作台、今日提醒、我的线索、公海线索、客户管理均通过同一组件打开弹窗。
   - 原因：避免复制 `CustomersView` 中的业务表单，确保查重和提交行为一致。

2. **客户创建保留在客户管理页**
   - `新增客户` 按钮统一导航到客户管理页并打开客户创建入口，或在客户管理页内直接打开客户创建弹窗。
   - 原因：客户创建当前只属于客户管理能力，跨页抽取会扩大范围；本轮重点是新建线索一致性。

3. **分页先采用前端派生**
   - 线索和用户列表使用当前 store/API 返回的数组，在前端搜索后分页。
   - 客户列表继续调用 `/customers?keyword=`，再对返回结果分页。
   - 原因：后端当前没有分页契约，前端分页可以满足当前模拟数据量和 UI 需求，并避免接口变更。

4. **列表体验以共享轻量模式统一，不强制抽通用大表格组件**
   - 使用相同 class、分页配置和派生逻辑，必要时可抽 `pagination` helper；避免一次性重构全部表格为复杂组件。
   - 原因：当前页面字段和操作差异较大，过早抽象会增加风险。

5. **抽屉阶段轨道只渲染历史到当前**
   - `LeadDetailPanel` 将阶段 rail 从 `ACTIVE_STAGES` 全量展示改为截断到当前阶段；当前阶段在标题区独立展示，顶部客户名右侧不再放阶段 tag。
   - 原因：满足用户“仅显示当前阶段、下方显示历史阶段到当前阶段”的明确要求。

## Risks / Trade-offs

- [前端分页不能处理服务端全量数据过大] → 当前接口本来返回数组，本轮不改变契约；后续如数据量扩大再提后端分页变更。
- [新增线索组件抽取可能影响客户管理既有测试] → 先补组件/页面测试，保持原有校验、查重和 API 调用断言。
- [视觉优化难以完全自动断言] → 对布局类变化记录非 TDD 例外，并用非无头浏览器 smoke 截图验证中文、抽屉、列表和弹窗。
- [多个页面同时改动回归面较大] → 运行受影响的视图测试、`vue-tsc`、构建和 OpenSpec strict validation。
